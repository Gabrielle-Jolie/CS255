# CS 255 Portfolio Reflection

### Briefly summarize the DriverPass project. Who was the client? What type of system did they want you to design?

The client for this project was DriverPass. They wanted a system that would help people study for their driver's license test. The system needed to let users take practice tests, schedule driving lessons, and access study materials. It also needed different access levels for students, instructors, and administrators.

### What did you do particularly well?

I think I did a good job figuring out what the client wanted and making sure those requirements were included in the design. I also liked working on the diagrams because they helped me understand how the system would work before any coding started.

### If you could choose one part of your work on these documents to revise, what would you pick? How would you improve it?

If I could go back, I would spend more time on the system design. There are a few parts I would explain better, and I would clean up my diagrams so they were easier to read. I think that would make the project easier for someone else to understand.

### How did you interpret the user's needs and implement them into your system design? Why is it so important to consider the user's needs when designing?

I looked at what the client expected the system to do and built the design around those requirements. I made sure different users had the features they needed based on their role. Thinking about the user's needs is important because if the system isn't easy to use or doesn't do what they need, it won't be successful.

### How do you approach designing software? What techniques or strategies would you use in the future to analyze and design a system?

I usually start by reading through the requirements and breaking everything into smaller parts. Then I think about how those parts fit together before designing the system. This project showed me that planning first saves time later. In future projects, I'll continue using requirements, diagrams, and planning documents before jumping into the coding.
